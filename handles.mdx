---
title: "Handles"
description: "The address that replaces account numbers, card numbers, and wallet IDs."
---

A handle looks like an email address, such as `alice@acme`, and works in a similar way: it is easy to remember, safe to share, and linked to a customer profile maintained by the issuing institution. Handles are used to send and receive money, as well as make online payments.

Handles are not sensitive and can be shared publicly. The only information a handle reveals is the institution that issued it. On its own, a handle cannot be used to make unauthorized charges or withdraw funds.

## What's actually in a handle

Two parts, split by `@`:

- **The user part** (`alice`) — chosen by the person, unique within their institution.
- **The domain part** (`acme`) — initially chosen by the institution that issued it.

A handle does not reveal a bank account number, card number, or wallet ID. Those details remain securely within the issuing institution, exactly as they do today. The handle is simply a reference that points to the underlying account.

## Only institutions issue handles

If you are a business collecting payments, you cannot issue handles. Your customers already have handles provided by their own bank or wallet provider. You simply collect their handle during checkout, or share your own handle when customers are paying directly from their banking app.

If you're an institution, here's what a handle actually holds:

| Field | What it's for |
| :-- | :-- |
| `handle` | The identifier itself, with the domain appended |
| `balance` | Used to check whether an outbound transaction is possible |
| `limit` | The maximum a single transaction from this handle can move |
| `code` | A one-time authorization code, rotated after every use |
| `status` | `active`, `restricted`, or `inactive` — controls whether the handle can transact at all |

## Creating a handle

```bash Create a handle
curl --request POST 'https://api.zelkea.com/handles' \
  --header 'Authorization: Bearer <API key>' \
  --header 'Content-Type: application/json' \
  --data-raw '{
    "handle": "alice", // domain is appended automatically
    "balance": 1000,
    "currency": "USD",
    "limit": 5000,
    "code": "XSO2B9",
    "status": "active"
  }'
```

```json Response
{
  "handle": "alice@acme",
  "balance": 1000,
  "currency": "USD",
  "limit": 5000,
  "status": "active",
  "created_at": "2026-07-10T00:21:49Z"
}
```

## Updating a handle

Balances change, limits get adjusted, accounts get frozen. Update only what changed:

```bash Freeze a handle
curl --request PATCH 'https://api.zelkea.com/handles/alice@acme' \
  --header 'Authorization: Bearer <API key>' \
  --header 'Content-Type: application/json' \
  --data-raw '{
    "status": "restricted"
  }'
```

<Warning>
  A handle's `code` resets automatically every time it's used to authorize a payment. If you're building a customer-facing app, make sure your UI always shows the _current_ code, not a cached one — an expired code will get rejected.
</Warning>

## Handles belong to the issuing institution

Handles are unique to the institution that issues them. The same person or business can have multiple handles if they use multiple banks or wallets connected to Zelkea. For example, Tom can have both `tom@goodwallet` and `tommy@lionbank`, each representing a different account relationship with a different institution.
